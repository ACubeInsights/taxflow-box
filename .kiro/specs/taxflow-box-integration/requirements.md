# Requirements Document

## Introduction

TaxFlow Pro requires a comprehensive backend API layer and Box Platform integration that orchestrates the full lifecycle of tax document management — from client onboarding and folder provisioning through document upload automation, review workflows, e-signatures, compliance enforcement, and AI-powered extraction. This spec covers the backend services, webhook pipelines, token strategies, and Box API integrations that sit between the existing box-wrapper-service (SDK init, JWT auth, schema sync, vault creation, metadata query) and the existing frontend UI (auth/RBAC dashboards, document workflow components). The system uses Box Content Cloud as the sole content platform, eliminating external databases for document storage and metadata, while enforcing HIPAA/IRS Pub 4557 compliance through Box-native security controls.

## Glossary

- **Onboarding_Service**: The backend service that automates client provisioning including App User creation, folder hierarchy setup, folder locks, and collaboration assignment
- **App_User**: A Box managed user (POST /users) created per client, enabling scoped token generation and isolated content access
- **Folder_Hierarchy**: The standard folder structure under each client root: Tax Year → {Tax, Uploads, SupportingDocs, SignedDocuments, InternalNotes}
- **Folder_Lock**: A Box folder lock (POST /folder_locks) preventing move or delete operations on protected folders
- **Collaboration**: A Box collaboration (POST /collaborations) granting a specific access role to a user on a folder
- **Token_Service**: The backend component managing Service Account tokens, App User tokens, and downscoped tokens via Box token exchange (POST /oauth2/token)
- **Downscoped_Token**: A token with reduced permissions generated via token exchange grant type, scoped to specific resources and actions
- **Permission_Matrix**: The mapping of folder types to allowed Box collaboration roles and token scopes
- **Webhook_Service**: The backend service that registers, verifies, and processes Box webhook events
- **Webhook_Signature**: HMAC-SHA256 signature included in Box webhook headers for payload verification
- **Post_Upload_Pipeline**: The automated sequence triggered after file upload: metadata application, task creation, notification dispatch
- **File_Request**: A Box file request (POST /file-requests/{id}/copy) enabling guest uploads to a specific folder without Box authentication
- **Metadata_Template**: The enterprise metadata template `taxflow_document` with fields: client_id, engagement_id, request_id, document_type, financial_year, status, reviewer, review_comments, reviewed_at, priority
- **Document_Status**: One of: pending_upload, uploaded, under_review, approved, revision_requested, waived, signed
- **Review_Service**: The backend service handling document approve, reject, waive, and bulk approve operations via Box metadata PATCH and task completion
- **Preview_Token**: A downscoped token with item_preview scope for Box Content Preview embedding
- **Internal_Note**: A file stored in the InternalNotes subfolder, accessible only to employees and admins
- **Revision_Flow**: The sequence: metadata update → comment creation → notification → client re-upload → webhook → status reset
- **Portal_Service**: The backend service providing client portal, employee dashboard, and CXO portfolio data via metadata queries
- **Metadata_Query**: A Box metadata query (POST /metadata_queries/execute_read) used to retrieve documents by metadata field values
- **Events_API**: The Box Events API (GET /events) used for detecting inactive clients and audit logging
- **Sign_Service**: The backend service integrating Box Sign for e-signature workflows
- **Sign_Request**: A Box Sign request (POST /sign_requests) with signer configuration, redirect URLs, and optional embedded signing
- **Deep_Link_Token**: A signed JWT generated by the backend for secure notification email links
- **Notification_Service**: The backend pipeline translating webhook events into email (SendGrid/SES) and in-app notifications
- **Retention_Policy**: A Box retention policy (POST /retention_policies) enforcing 7-year document retention with non_modifiable disposition
- **Legal_Hold**: A Box legal hold applied to files or folders preventing deletion during legal proceedings
- **Security_Classification**: Box metadata classification (POST /files/{id}/metadata/enterprise/securityClassification) for data sensitivity labeling
- **AI_Extraction_Service**: The backend service using Box AI structured extraction (POST /ai/extract_structured) and validation (POST /ai/ask)
- **AI_Agent**: A custom Box AI agent (POST /ai/agents) configured for TaxFlow-specific document analysis
- **Cache_Layer**: In-memory LRU cache with TTL support for metadata query results and token storage
- **Chunked_Upload**: Box chunked upload API for files exceeding 50MB
- **Rate_Limiter**: In-memory priority queue managing Box API request concurrency
- **Circuit_Breaker**: Fault tolerance pattern that stops requests to Box API during sustained failures and recovers gracefully

## Requirements

### Requirement 1: Client App User Provisioning

**User Story:** As a system administrator, I want each new client to receive a dedicated Box App User, so that content access is isolated per client and scoped tokens can be generated.

#### Acceptance Criteria

1. WHEN a new client is onboarded, THE Onboarding_Service SHALL create a Box App User via POST /users with `is_platform_access_only: true` and the client display name
2. WHEN the App_User is created, THE Onboarding_Service SHALL store the returned `user_id` in the client onboarding response for downstream token generation
3. THE Onboarding_Service SHALL set the App_User `space_amount` to the configured storage quota (default 10GB)
4. IF the POST /users call returns an HTTP 409 conflict, THEN THE Onboarding_Service SHALL retrieve the existing App_User by login email and return the existing user_id
5. IF the POST /users call fails with a non-409 error, THEN THE Onboarding_Service SHALL return a descriptive error including the HTTP status code and Box error message

### Requirement 2: Client Folder Hierarchy Creation

**User Story:** As a system administrator, I want a standardized folder hierarchy created for each client, so that documents are organized consistently across all engagements.

#### Acceptance Criteria

1. WHEN a client is onboarded, THE Onboarding_Service SHALL create a root folder named `{clientName} ({externalId})` under the configured Clients root folder via POST /folders
2. WHEN the root folder is created, THE Onboarding_Service SHALL create a tax year subfolder named with the current financial year (e.g., "2025") under the root folder
3. WHEN the tax year folder is created, THE Onboarding_Service SHALL create exactly five subfolders: "Tax", "Uploads", "SupportingDocs", "SignedDocuments", "InternalNotes"
4. WHEN all folders are created, THE Onboarding_Service SHALL return a manifest object mapping each folder type to its Box folder ID
5. IF any folder creation fails after partial hierarchy creation, THEN THE Onboarding_Service SHALL return an error including the IDs of successfully created folders for manual remediation
6. THE Onboarding_Service SHALL execute folder creation sequentially (root → year → subfolders) to ensure parent folders exist before children

### Requirement 3: Folder Lock Enforcement

**User Story:** As a compliance officer, I want critical folders locked against move and delete operations, so that the folder structure integrity is maintained.

#### Acceptance Criteria

1. WHEN the Folder_Hierarchy is created, THE Onboarding_Service SHALL apply a Folder_Lock via POST /folder_locks to the client root folder
2. WHEN the Folder_Hierarchy is created, THE Onboarding_Service SHALL apply a Folder_Lock via POST /folder_locks to the "SignedDocuments" subfolder
3. WHEN the Folder_Hierarchy is created, THE Onboarding_Service SHALL apply a Folder_Lock via POST /folder_locks to the "InternalNotes" subfolder
4. IF a Folder_Lock creation fails, THEN THE Onboarding_Service SHALL log the failure with the folder ID and continue processing remaining locks without aborting the onboarding

### Requirement 4: Collaboration Setup and Permission Matrix

**User Story:** As a system administrator, I want collaborations automatically assigned per folder type with role-appropriate permissions, so that clients, employees, and admins have correct access from day one.

#### Acceptance Criteria

1. WHEN the Folder_Hierarchy is created, THE Onboarding_Service SHALL create a Collaboration via POST /collaborations granting the App_User "viewer_uploader" role on the "Uploads" folder
2. WHEN the Folder_Hierarchy is created, THE Onboarding_Service SHALL create a Collaboration granting the App_User "viewer" role on the "Tax" folder
3. WHEN the Folder_Hierarchy is created, THE Onboarding_Service SHALL create a Collaboration granting the App_User "viewer" role on the "SignedDocuments" folder
4. THE Onboarding_Service SHALL NOT create any Collaboration granting the App_User access to the "InternalNotes" folder
5. WHEN the Folder_Hierarchy is created, THE Onboarding_Service SHALL create a Collaboration granting the assigned employee "editor" role on the client root folder
6. IF a Collaboration creation returns HTTP 409 (already exists), THEN THE Onboarding_Service SHALL treat the conflict as a successful state and continue
7. THE Permission_Matrix SHALL enforce the following role mapping: Uploads → viewer_uploader (client), Tax → editor (employee) + viewer (client), SupportingDocs → editor (employee) + viewer_uploader (client), SignedDocuments → viewer (client) + viewer (employee), InternalNotes → editor (employee only)

### Requirement 5: Token Strategy — Service Account and App User Tokens

**User Story:** As a backend service, I want to generate appropriately scoped tokens for different operations, so that each API call uses the minimum required privilege level.

#### Acceptance Criteria

1. THE Token_Service SHALL generate Service Account tokens for administrative operations (folder creation, webhook registration, metadata template management)
2. THE Token_Service SHALL generate App User tokens via `POST /oauth2/token` with `grant_type: urn:ietf:params:oauth:grant-type:jwt-bearer` and `subject_type: user` for client-scoped operations
3. THE Token_Service SHALL cache generated tokens in the Cache_Layer with TTL set to 90% of the token expiry duration to prevent expired token usage
4. WHEN a cached token is within 10% of its expiry duration, THE Token_Service SHALL proactively refresh the token before the next request
5. IF token generation fails, THEN THE Token_Service SHALL retry once after a 1-second delay before returning an error

### Requirement 6: Downscoped Token Generation

**User Story:** As a backend service, I want to generate downscoped tokens restricted to specific folders and actions, so that frontend clients receive minimum-privilege access tokens.

#### Acceptance Criteria

1. WHEN a downscoped token is requested, THE Token_Service SHALL call POST /oauth2/token with `grant_type: urn:ietf:params:oauth:grant-type:token-exchange`, `subject_token` set to the parent App User token, and the requested `scope` parameter
2. THE Token_Service SHALL support the following scope values: `item_preview`, `item_download`, `item_upload`, `item_readwrite`
3. WHEN a downscoped token is requested with a `resource` parameter, THE Token_Service SHALL include the Box folder or file URL in the token exchange request to restrict the token to that specific resource
4. THE Token_Service SHALL return the downscoped token with its `expires_in` value to the caller
5. IF the token exchange call fails, THEN THE Token_Service SHALL return a descriptive error including the requested scope and resource

### Requirement 7: Webhook Registration and Management

**User Story:** As a system administrator, I want webhooks automatically registered on client folders, so that file events trigger backend automation without polling.

#### Acceptance Criteria

1. WHEN a client is onboarded, THE Webhook_Service SHALL register a webhook via POST /webhooks on the client root folder with triggers: FILE.UPLOADED, FILE.DELETED, FILE.MOVED
2. THE Webhook_Service SHALL store the webhook ID and the primary and secondary signature keys returned by Box for signature verification
3. WHEN a webhook is registered, THE Webhook_Service SHALL set the `address` field to the configured webhook endpoint URL (e.g., `https://{domain}/api/webhooks/box`)
4. IF webhook registration fails with HTTP 409 (already exists), THEN THE Webhook_Service SHALL retrieve the existing webhook for the folder and verify its trigger configuration
5. IF webhook registration fails with a non-409 error, THEN THE Webhook_Service SHALL log the error and schedule a retry via the Rate_Limiter queue

### Requirement 8: Webhook Signature Verification

**User Story:** As a security engineer, I want all incoming webhook payloads verified via HMAC-SHA256 signatures, so that only authentic Box events are processed.

#### Acceptance Criteria

1. WHEN a webhook payload is received, THE Webhook_Service SHALL extract the `BOX-SIGNATURE-PRIMARY` and `BOX-SIGNATURE-SECONDARY` headers
2. THE Webhook_Service SHALL compute HMAC-SHA256 of the raw request body using the stored primary signature key and compare it to the `BOX-SIGNATURE-PRIMARY` header value
3. IF the primary signature matches, THE Webhook_Service SHALL accept the payload for processing
4. IF the primary signature does not match, THE Webhook_Service SHALL compute HMAC-SHA256 using the secondary signature key and compare it to the `BOX-SIGNATURE-SECONDARY` header value
5. IF neither signature matches, THEN THE Webhook_Service SHALL reject the payload with HTTP 403 and log the verification failure with the webhook ID and source IP
6. THE Webhook_Service SHALL use constant-time string comparison for signature verification to prevent timing attacks

### Requirement 9: Post-Upload Automation Pipeline

**User Story:** As a tax preparer, I want uploaded documents to be automatically tagged with metadata and assigned for review, so that no manual triage is needed.

#### Acceptance Criteria

1. WHEN a FILE.UPLOADED webhook event is received for a file in the "Uploads" folder, THE Post_Upload_Pipeline SHALL apply the `taxflow_document` Metadata_Template to the file via POST /files/{file_id}/metadata/enterprise/taxflow_document
2. WHEN metadata is applied, THE Post_Upload_Pipeline SHALL set the following fields: `client_id` (from parent folder metadata), `status` set to "uploaded", `financial_year` (from parent year folder name), `priority` set to "normal"
3. WHEN metadata is applied, THE Post_Upload_Pipeline SHALL create a Box task via POST /tasks with `action: review`, `message: "Review uploaded document"`, and `completion_rule: all_assignees`
4. WHEN the task is created, THE Post_Upload_Pipeline SHALL assign the task to the designated reviewer via POST /tasks/{task_id}/assignments
5. WHEN the task assignment is complete, THE Notification_Service SHALL dispatch a notification to the assigned reviewer
6. IF metadata application fails, THEN THE Post_Upload_Pipeline SHALL queue the file ID for retry via the Rate_Limiter and log the failure

### Requirement 10: Taxflow Document Metadata Template Schema

**User Story:** As a system architect, I want a comprehensive metadata template that captures all document lifecycle fields, so that the system can query, filter, and report on documents without an external database.

#### Acceptance Criteria

1. THE Metadata_Template `taxflow_document` SHALL define the following fields: `client_id` (string), `engagement_id` (string), `request_id` (string), `document_type` (string), `financial_year` (string), `status` (enum: pending_upload, uploaded, under_review, approved, revision_requested, waived, signed), `reviewer` (string), `review_comments` (string), `reviewed_at` (date), `priority` (enum: low, normal, high, urgent)
2. THE Metadata_Template SHALL be created in the `enterprise` scope with `templateKey: taxflow_document`
3. THE `client_id`, `engagement_id`, and `status` fields SHALL be configured as queryable (indexed) to support Metadata_Query operations
4. IF the template already exists (HTTP 409), THEN THE system SHALL verify the existing template contains all required fields and log any discrepancies

### Requirement 11: File Request Creation for Guest Uploads

**User Story:** As a tax preparer, I want to generate secure file request links for clients who do not have Box accounts, so that guests can upload documents directly to the correct folder.

#### Acceptance Criteria

1. WHEN a file request is created for a client engagement, THE Onboarding_Service SHALL call POST /file-requests/{template_id}/copy with the target "Uploads" folder ID and a descriptive title
2. THE file request SHALL include an `expires_at` date set to the engagement due date plus a 7-day grace period
3. THE file request SHALL set `is_email_required` to true and `is_description_required` to false
4. WHEN the file request is created, THE Onboarding_Service SHALL return the `url` field for distribution to the client
5. IF the file request copy fails, THEN THE Onboarding_Service SHALL return a descriptive error including the template ID and target folder ID

### Requirement 12: Document Approval Workflow

**User Story:** As a tax preparer, I want to approve a document with a single action that updates metadata and completes the review task, so that the approval is recorded consistently.

#### Acceptance Criteria

1. WHEN an employee approves a document, THE Review_Service SHALL PATCH the file metadata at /files/{file_id}/metadata/enterprise/taxflow_document with: `status` set to "approved", `reviewer` set to the employee identifier, `reviewed_at` set to the current ISO 8601 timestamp, `review_comments` cleared
2. WHEN metadata is updated, THE Review_Service SHALL complete the associated Box task by updating the task assignment status to "completed" via PUT /tasks/{task_id}/assignments/{assignment_id}
3. WHEN the task is completed, THE Notification_Service SHALL dispatch an approval notification to the client
4. THE Review_Service SHALL use JSON Patch operations (op: "replace", op: "add") for metadata updates to ensure atomic field changes
5. IF the metadata PATCH fails, THEN THE Review_Service SHALL return an error and NOT complete the task assignment to maintain consistency

### Requirement 13: Document Rejection Workflow

**User Story:** As a tax preparer, I want to reject a document with comments that are visible to the client, so that the client knows exactly what to fix.

#### Acceptance Criteria

1. WHEN an employee rejects a document, THE Review_Service SHALL PATCH the file metadata with: `status` set to "revision_requested", `reviewer` set to the employee identifier, `reviewed_at` set to the current ISO 8601 timestamp, `review_comments` set to the employee-provided rejection reason
2. WHEN metadata is updated, THE Review_Service SHALL create a comment on the file via POST /comments with the rejection reason as the `message` and `tagged_message` referencing the client
3. WHEN the comment is created, THE Notification_Service SHALL dispatch a revision-requested notification to the client including the rejection reason
4. THE Review_Service SHALL NOT complete the Box task assignment on rejection — the task remains open for re-review after client re-upload
5. IF the rejection reason is empty or missing, THEN THE Review_Service SHALL return HTTP 400 with a validation error requiring a non-empty rejection reason

### Requirement 14: Document Waive Workflow

**User Story:** As a tax preparer, I want to waive a document requirement, so that the engagement can proceed without a specific document when it is not applicable.

#### Acceptance Criteria

1. WHEN an employee waives a document requirement, THE Review_Service SHALL PATCH the file metadata with: `status` set to "waived", `reviewer` set to the employee identifier, `reviewed_at` set to the current ISO 8601 timestamp, `review_comments` set to the waive reason
2. WHEN a document is waived, THE Review_Service SHALL complete the associated Box task assignment with status "completed"
3. THE Notification_Service SHALL dispatch a waive notification to the client

### Requirement 15: Bulk Approve Operations

**User Story:** As a tax preparer, I want to approve multiple documents at once, so that I can efficiently process batches of reviewed documents.

#### Acceptance Criteria

1. WHEN a bulk approve request is received with an array of file IDs, THE Review_Service SHALL execute the approval workflow (Requirement 12) for each file in the array
2. THE Review_Service SHALL process bulk approvals concurrently with a maximum concurrency limit of 5 parallel operations to respect Box API rate limits
3. WHEN all approvals complete, THE Review_Service SHALL return a summary indicating the count of successful approvals and any failures with their file IDs and error messages
4. IF any individual approval fails, THEN THE Review_Service SHALL continue processing remaining files and include the failure in the summary response

### Requirement 16: Box Content Preview Token Generation

**User Story:** As a frontend developer, I want to obtain a scoped preview token for a specific file, so that Box Content Preview can be embedded securely in the review UI.

#### Acceptance Criteria

1. WHEN a preview token is requested for a file, THE Token_Service SHALL generate a Downscoped_Token with scope `item_preview` and `resource` set to the file URL (`https://api.box.com/2.0/files/{file_id}`)
2. THE preview token SHALL have a maximum TTL of 60 minutes
3. THE Token_Service SHALL return the preview token and its expiry timestamp to the caller
4. IF the file ID does not exist or the caller lacks access, THEN THE Token_Service SHALL return HTTP 404 or HTTP 403 respectively

### Requirement 17: Internal Notes Management

**User Story:** As a tax preparer, I want to create and manage internal notes that are invisible to clients, so that I can document internal observations and decisions.

#### Acceptance Criteria

1. WHEN an employee creates an internal note, THE Review_Service SHALL upload the note content as a file to the client's "InternalNotes" subfolder via the Box upload API
2. THE internal note file SHALL be named with the pattern `{timestamp}_{author}_{subject}.txt`
3. WHEN an internal note is created, THE Review_Service SHALL apply the `taxflow_document` metadata with `document_type` set to "internal_note" and `status` set to "approved"
4. THE Permission_Matrix SHALL ensure the "InternalNotes" folder has no client Collaboration, restricting access to employees and admins only
5. WHEN internal notes are listed, THE Review_Service SHALL return notes sorted by creation date descending

### Requirement 18: Revision Flow — Re-upload and Status Reset

**User Story:** As a system, I want client re-uploads after rejection to automatically reset the document status and notify the reviewer, so that the revision cycle is seamless.

#### Acceptance Criteria

1. WHEN a FILE.UPLOADED webhook event is received for a file in a folder where a previous file has `status: revision_requested`, THE Post_Upload_Pipeline SHALL update the existing file metadata `status` to "uploaded" and clear `review_comments`
2. WHEN the status is reset, THE Post_Upload_Pipeline SHALL create a new Box task assignment for the original reviewer
3. WHEN the task is assigned, THE Notification_Service SHALL dispatch a re-upload notification to the reviewer including the document name and client identifier
4. THE Post_Upload_Pipeline SHALL add a comment on the new file via POST /comments indicating "Re-uploaded by client — pending re-review"

### Requirement 19: Client Portal — Progress Tracking via Metadata Queries

**User Story:** As a client, I want to see the real-time status of all my document requests, so that I know what has been submitted, what is pending, and what needs revision.

#### Acceptance Criteria

1. WHEN a client requests their document progress, THE Portal_Service SHALL execute a Metadata_Query via POST /metadata_queries/execute_read with `from: enterprise_<id>.taxflow_document`, filtering by `client_id` equal to the requesting client's identifier
2. THE Portal_Service SHALL return documents grouped by `status` with counts for each Document_Status value
3. THE Portal_Service SHALL include for each document: file name, document_type, status, priority, reviewed_at, and review_comments (if status is revision_requested)
4. THE Portal_Service SHALL cache Metadata_Query results in the Cache_Layer with a TTL of 60 seconds to reduce Box API calls
5. WHEN the cached result is stale, THE Portal_Service SHALL execute a fresh Metadata_Query and update the cache

### Requirement 20: Employee Dashboard — Pending Reviews and Overdue Items

**User Story:** As a tax preparer, I want to see all documents pending my review and any overdue items, so that I can prioritize my workload effectively.

#### Acceptance Criteria

1. WHEN an employee requests their dashboard data, THE Portal_Service SHALL execute a Metadata_Query filtering by `reviewer` equal to the employee identifier and `status` in ("uploaded", "under_review")
2. THE Portal_Service SHALL return pending review items sorted by `priority` (urgent first) then by upload date (oldest first)
3. THE Portal_Service SHALL flag items as overdue when the associated engagement due date has passed
4. THE Portal_Service SHALL return a document checklist per client showing total required documents, submitted count, approved count, and pending count
5. THE Portal_Service SHALL cache employee dashboard results in the Cache_Layer with a TTL of 30 seconds

### Requirement 21: CXO Portfolio View — Cross-Client Aggregation

**User Story:** As a CXO/Partner, I want a portfolio-level view aggregating document status across all clients, so that I can monitor firm-wide compliance and identify bottlenecks.

#### Acceptance Criteria

1. WHEN a CXO requests portfolio data, THE Portal_Service SHALL execute Metadata_Queries across all clients to aggregate document counts by status
2. THE Portal_Service SHALL return per-client summary rows containing: client name, total documents, approved count, pending count, revision count, and overall completion percentage
3. THE Portal_Service SHALL return firm-wide totals: total clients, total documents, overall compliance rate (approved / total × 100), and count of clients with overdue items
4. THE Portal_Service SHALL use marker-based pagination when querying across large client sets, processing results in batches of 100
5. THE Portal_Service SHALL cache CXO portfolio results in the Cache_Layer with a TTL of 120 seconds

### Requirement 22: Inactive Client Detection via Events API

**User Story:** As a system administrator, I want to detect clients with no activity for a configurable period, so that inactive engagements can be flagged for follow-up.

#### Acceptance Criteria

1. THE Portal_Service SHALL query the Box Events API via GET /events with `stream_type: admin_logs` and `event_type: UPLOAD,PREVIEW,DOWNLOAD` to retrieve recent activity
2. THE Portal_Service SHALL compare the latest event timestamp per client folder against the configured inactivity threshold (default 30 days)
3. WHEN a client has no events within the inactivity threshold, THE Portal_Service SHALL flag the client as inactive in the response
4. THE Portal_Service SHALL support configurable inactivity thresholds via environment variable `INACTIVE_THRESHOLD_DAYS`
5. THE Portal_Service SHALL use the `stream_position` parameter for marker-based pagination when processing large event streams

### Requirement 23: File Version History Retrieval

**User Story:** As a tax preparer, I want to view the version history of a document, so that I can track changes and compare revisions.

#### Acceptance Criteria

1. WHEN a version history is requested, THE Portal_Service SHALL call GET /files/{file_id}/versions to retrieve all file versions
2. THE Portal_Service SHALL return each version with: version_id, name, size, modified_by (user name), modified_at timestamp, and version number
3. THE Portal_Service SHALL return versions sorted by version number descending (most recent first)
4. IF the file has no previous versions, THEN THE Portal_Service SHALL return the current version as the only entry

### Requirement 24: Report Generation and Zip Downloads

**User Story:** As a tax preparer, I want to download a zip archive of selected documents, so that I can export engagement files for offline review or client delivery.

#### Acceptance Criteria

1. WHEN a zip download is requested with an array of file IDs, THE Portal_Service SHALL call POST /zip_downloads with the file items array
2. THE Portal_Service SHALL poll the returned `status_url` until the zip download status is "success"
3. WHEN the zip is ready, THE Portal_Service SHALL return the `download_url` to the caller
4. IF the zip download status is "failed", THEN THE Portal_Service SHALL return an error with the failure reason from the Box response
5. THE Portal_Service SHALL enforce a maximum of 100 files per zip download request and return HTTP 400 if exceeded

### Requirement 25: Box Sign Integration — Signature Request Creation

**User Story:** As a tax preparer, I want to send documents for e-signature via Box Sign, so that clients can sign tax documents digitally within the platform.

#### Acceptance Criteria

1. WHEN a signature request is initiated, THE Sign_Service SHALL call POST /sign_requests with: `source_files` containing the file ID, `signers` array with client email and role "signer", `parent_folder` set to the "SignedDocuments" folder ID
2. THE Sign_Service SHALL set `redirect_url` to the configured client portal URL with the engagement context
3. THE Sign_Service SHALL set `declined_redirect_url` to the configured decline handling URL
4. WHEN `is_embedded` is true in the request, THE Sign_Service SHALL return the `embed_url` from the Box Sign response for in-app signing
5. WHEN the sign request is created, THE Sign_Service SHALL update the file metadata `status` to "pending_signature" via PATCH
6. IF the POST /sign_requests call fails, THEN THE Sign_Service SHALL return a descriptive error including the file ID and signer email

### Requirement 26: Box Sign Webhook Event Processing

**User Story:** As a system, I want to automatically process signature completion, decline, and expiry events, so that document status and notifications stay current.

#### Acceptance Criteria

1. WHEN a SIGN_REQUEST.COMPLETED webhook event is received, THE Sign_Service SHALL update the file metadata `status` to "signed" and set `reviewed_at` to the completion timestamp
2. WHEN a SIGN_REQUEST.COMPLETED event is received, THE Sign_Service SHALL move the signed document to the "SignedDocuments" folder if not already there
3. WHEN a SIGN_REQUEST.DECLINED webhook event is received, THE Sign_Service SHALL update the file metadata `status` to "revision_requested" and set `review_comments` to "Signature declined by signer"
4. WHEN a SIGN_REQUEST.EXPIRED webhook event is received, THE Sign_Service SHALL update the file metadata `status` to "pending_upload" and dispatch an expiry notification to the employee
5. THE Notification_Service SHALL dispatch appropriate notifications for each sign event type to both the employee and client

### Requirement 27: Deep-Link Token Mechanism for Notifications

**User Story:** As a client receiving an email notification, I want to click a link that takes me directly to the relevant document, so that I can act on notifications without manual navigation.

#### Acceptance Criteria

1. WHEN a notification is generated, THE Notification_Service SHALL create a Deep_Link_Token as a signed JWT containing: `file_id`, `client_id`, `action` (view, upload, sign), and `exp` (expiry timestamp)
2. THE Deep_Link_Token SHALL be signed using HMAC-SHA256 with a server-side secret key configured via environment variable `DEEP_LINK_SECRET`
3. THE Deep_Link_Token SHALL expire after 72 hours from generation
4. WHEN a deep link is accessed, THE backend SHALL verify the JWT signature and expiry before redirecting to the appropriate frontend route with context parameters
5. IF the Deep_Link_Token is expired or has an invalid signature, THEN THE backend SHALL return HTTP 401 with a message indicating the link has expired

### Requirement 28: Notification Pipeline — Event Translation and Delivery

**User Story:** As a system, I want webhook events translated into user-friendly notifications delivered via email and in-app channels, so that all stakeholders are informed of document lifecycle changes.

#### Acceptance Criteria

1. WHEN a webhook event is processed, THE Notification_Service SHALL translate the Box event into a business event (e.g., FILE.UPLOADED → "New document uploaded by {client_name}")
2. THE Notification_Service SHALL support two delivery channels: email (via SendGrid or SES) and in-app notification storage
3. WHEN an email notification is dispatched, THE Notification_Service SHALL include the Deep_Link_Token URL for direct document access
4. THE Notification_Service SHALL use configurable email templates per event type (upload, approval, rejection, signature request, signature complete)
5. IF email delivery fails, THEN THE Notification_Service SHALL queue the notification for retry (maximum 3 retries with exponential backoff) and log the failure
6. THE Notification_Service SHALL store in-app notifications in memory with: recipient_id, event_type, message, document_reference, read status, and created_at timestamp. Notifications are ephemeral and reset on server restart.

### Requirement 29: 7-Year Retention Policy Enforcement

**User Story:** As a compliance officer, I want all tax documents subject to a 7-year retention policy, so that the firm meets IRS Pub 4557 record-keeping requirements.

#### Acceptance Criteria

1. WHEN the system initializes, THE Compliance_Service SHALL ensure a retention policy exists via POST /retention_policies with `policy_name: "TaxFlow 7-Year Retention"`, `policy_type: finite`, `retention_length: 2555` (7 years in days), and `disposition_action: permanently_delete`
2. THE retention policy SHALL be created with `are_owners_notified: true` and `can_owner_extend_retention: false` to enforce non_modifiable disposition

### Requirement 35: In-Memory Caching for Metadata Queries and Tokens

**User Story:** As a backend service, I want frequently accessed metadata query results and tokens cached in memory, so that response times are fast and Box API rate limits are respected.

#### Acceptance Criteria

1. THE Cache_Layer SHALL use an in-memory LRU cache with configurable maximum entry count (default 1000) and per-entry TTL support
2. THE Cache_Layer SHALL cache Metadata_Query results with keys in the format `mq:{query_hash}` and configurable TTL per query type
3. THE Cache_Layer SHALL cache tokens with keys in the format `token:{user_id}:{scope}` and TTL set to 90% of the token expiry
4. WHEN a cache entry is requested and exists, THE Cache_Layer SHALL return the cached value without calling the Box API
5. WHEN a cache entry is requested and does not exist or is expired, THE Cache_Layer SHALL execute the Box API call, store the result, and return it
6. THE Cache_Layer SHALL automatically evict expired entries and entries exceeding the maximum count using least-recently-used eviction

### Requirement 37: Rate Limiting and Request Queuing

**User Story:** As a backend service, I want API requests queued and rate-limited, so that the system respects Box API rate limits and handles burst traffic gracefully.

### Requirement 38: Circuit Breaker for Box API Failures

**User Story:** As a backend service, I want a circuit breaker that stops requests to Box during sustained failures, so that the system degrades gracefully and recovers automatically.

#### Acceptance Criteria

1. THE Circuit_Breaker SHALL track Box API failure rates over a rolling 60-second window
2. WHEN the failure rate exceeds 50% with a minimum of 10 requests in the window, THE Circuit_Breaker SHALL transition to the "open" state and reject all Box API requests with HTTP 503
3. WHILE the Circuit_Breaker is in the "open" state, THE Circuit_Breaker SHALL wait for a configurable cooldown period (default 30 seconds) before transitioning to "half-open"
4. WHILE the Circuit_Breaker is in the "half-open" state, THE Circuit_Breaker SHALL allow a single probe request through — if it succeeds, transition to "closed"; if it fails, transition back to "open"
5. THE Circuit_Breaker SHALL emit state change events (closed → open, open → half-open, half-open → closed) for monitoring and alerting
