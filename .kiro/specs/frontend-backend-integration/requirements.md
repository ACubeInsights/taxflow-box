# Requirements Document

## Introduction

This feature wires the existing TaxFlow Pro React frontend (`taxflow-app`) to the existing backend API (`taxflow-api`) so that all UI interactions use real Box data instead of mock/local state. The frontend currently relies on hardcoded mock data in `DocumentWorkflowContext`, local state machine transitions, and static arrays in dashboard components. The backend already has 15 service modules with 8 route files calling real Box APIs. This integration replaces every mock data path with live API calls, adds proper loading/error states, and connects the demo auth system to real Box App User tokens.

## Glossary

- **API_Service_Layer**: The `taxflow-app/src/services/api.js` module that provides typed methods for all backend HTTP endpoints, handling request construction, response parsing, and error normalization
- **DocumentWorkflow_Context**: The React context (`DocumentWorkflowContext.jsx`) that manages document request state, vault initialization, and workflow transitions for the entire frontend
- **Client_Dashboard**: The React component displaying client progress, preparer requests, upload dropzone, and document request list for logged-in clients
- **Employee_Dashboard**: The React component displaying assigned clients, pending reviews, AI insights, and document request management for tax preparers
- **CXO_Dashboard**: The React component displaying firm-wide portfolio compliance, filing progress, and critical alerts for executive users
- **SuperAdmin_Dashboard**: The React component displaying system health, user management, and Box AI integration status for administrators
- **Upload_Dropzone**: The drag-and-drop file upload component that sends files to Box via the backend upload endpoint
- **Review_Mode**: The split-pane layout where employees preview documents and take approve/reject/waive actions
- **Request_Creator**: The slide-out drawer used by employees to create new document requests that persist via the backend
- **Auth_Context**: The React context (`AuthContext.jsx`) managing user identity, role selection, and authentication tokens
- **Loading_State**: A visual indicator (spinner, skeleton, or progress bar) displayed while an API call is in flight
- **Error_State**: A visual indicator (error message with optional retry button) displayed when an API call fails
- **Backend_API**: The Express.js server (`taxflow-api`) running on port 3001 with routes for onboarding, portal, reviews, tokens, sign, notifications, compliance, and webhooks
- **Box_App_User_Token**: A scoped access token generated by the backend `TokenService` for a specific Box App User, used to authenticate frontend API requests
- **Preview_Token**: A downscoped Box token with `item_preview` scope, used to embed Box Content Preview in the frontend

## Requirements

### Requirement 1: API Service Layer — Core HTTP Client

**User Story:** As a frontend developer, I want a centralized API service module with methods for all backend endpoints, so that components can call the backend without constructing raw HTTP requests.

#### Acceptance Criteria

1. THE API_Service_Layer SHALL export named API objects (`onboardingApi`, `portalApi`, `reviewApi`, `tokenApi`, `signApi`, `notificationApi`, `complianceApi`) each containing async methods that map to the corresponding Backend_API route
2. WHEN any API method receives a non-2xx HTTP response, THE API_Service_Layer SHALL parse the response body for an `error` field and throw an Error with that message
3. WHEN any API method encounters a network failure, THE API_Service_Layer SHALL throw an Error with a descriptive message including the endpoint path
4. THE API_Service_Layer SHALL read the base URL from the `VITE_API_URL` environment variable, falling back to `http://localhost:3001/api`
5. WHEN an authentication token is available in Auth_Context, THE API_Service_Layer SHALL include the token in an `Authorization: Bearer <token>` header on every request
6. THE API_Service_Layer SHALL set `Content-Type: application/json` for all non-upload requests and omit the header for `multipart/form-data` upload requests

### Requirement 2: API Service Layer — Onboarding Endpoint

**User Story:** As a frontend developer, I want an API method for client onboarding, so that vault initialization calls the real backend.

#### Acceptance Criteria

1. THE API_Service_Layer SHALL expose an `onboardingApi.onboardClient(clientName, externalId, email, employeeEmail, financialYear)` method that sends a POST request to `/api/onboarding`
2. WHEN the POST request succeeds with status 201, THE `onboardClient` method SHALL return the parsed `OnboardingResult` object containing `appUser`, `folders`, `locks`, `collaborations`, and `webhookId`
3. IF the POST request returns status 400, THEN THE `onboardClient` method SHALL throw an Error containing the missing field names from the response

### Requirement 3: API Service Layer — Portal Endpoints

**User Story:** As a frontend developer, I want API methods for all portal data endpoints, so that dashboards can fetch real data from Box.

#### Acceptance Criteria

1. THE API_Service_Layer SHALL expose a `portalApi.getClientProgress(clientId)` method that sends a GET request to `/api/portal/client/{clientId}/progress`
2. THE API_Service_Layer SHALL expose a `portalApi.getEmployeeDashboard(employeeId)` method that sends a GET request to `/api/portal/employee/{employeeId}/dashboard`
3. THE API_Service_Layer SHALL expose a `portalApi.getCXOPortfolio(cursor, limit)` method that sends a GET request to `/api/portal/cxo/portfolio` with optional `cursor` and `limit` query parameters
4. THE API_Service_Layer SHALL expose a `portalApi.getInactiveClients(thresholdDays)` method that sends a GET request to `/api/portal/inactive-clients` with an optional `thresholdDays` query parameter
5. THE API_Service_Layer SHALL expose a `portalApi.getFileVersions(fileId)` method that sends a GET request to `/api/portal/files/{fileId}/versions`
6. THE API_Service_Layer SHALL expose a `portalApi.createZipDownload(fileIds)` method that sends a POST request to `/api/portal/zip-download` with a JSON body containing the `fileIds` array

### Requirement 4: API Service Layer — Review Endpoints

**User Story:** As a frontend developer, I want API methods for document review actions, so that approve/reject/waive operations persist to Box.

#### Acceptance Criteria

1. THE API_Service_Layer SHALL expose a `reviewApi.approve(fileId, employeeId)` method that sends a POST request to `/api/reviews/{fileId}/approve`
2. THE API_Service_Layer SHALL expose a `reviewApi.reject(fileId, employeeId, reason)` method that sends a POST request to `/api/reviews/{fileId}/reject`
3. THE API_Service_Layer SHALL expose a `reviewApi.waive(fileId, employeeId, reason)` method that sends a POST request to `/api/reviews/{fileId}/waive`
4. THE API_Service_Layer SHALL expose a `reviewApi.bulkApprove(fileIds, employeeId)` method that sends a POST request to `/api/reviews/bulk-approve`
5. THE API_Service_Layer SHALL expose a `reviewApi.createNote(clientFolderId, author, subject, content)` method that sends a POST request to `/api/reviews/{clientFolderId}/notes`
6. THE API_Service_Layer SHALL expose a `reviewApi.listNotes(clientFolderId)` method that sends a GET request to `/api/reviews/{clientFolderId}/notes`

### Requirement 5: API Service Layer — Token, Sign, Notification, and Compliance Endpoints

**User Story:** As a frontend developer, I want API methods for tokens, e-signatures, notifications, and compliance, so that all backend capabilities are accessible from the frontend.

#### Acceptance Criteria

1. THE API_Service_Layer SHALL expose a `tokenApi.getPreviewToken(fileId, userId)` method that sends a POST request to `/api/tokens/preview`
2. THE API_Service_Layer SHALL expose a `signApi.createSignRequest(fileId, signerEmail, signedDocsFolderId, isEmbedded)` method that sends a POST request to `/api/sign/request`
3. THE API_Service_Layer SHALL expose a `notificationApi.getNotifications(recipientId)` method that sends a GET request to `/api/notifications/{recipientId}`
4. THE API_Service_Layer SHALL expose a `complianceApi.classify(fileId, level)` method that sends a POST request to `/api/compliance/classify/{fileId}`

### Requirement 6: DocumentWorkflow Context — Replace Mock State with API Calls

**User Story:** As a frontend developer, I want the DocumentWorkflow_Context to fetch and mutate data via the backend API, so that all document state reflects real Box data.

#### Acceptance Criteria

1. WHEN `initializeVault` is called, THE DocumentWorkflow_Context SHALL call `onboardingApi.onboardClient` with the provided client name, external ID, and email, and store the returned `OnboardingResult` as the vault state
2. WHEN `initializeVault` succeeds, THE DocumentWorkflow_Context SHALL call `portalApi.getClientProgress` with the client ID and populate the `requests` state from the returned documents array
3. WHEN the `APPROVE` action is dispatched, THE DocumentWorkflow_Context SHALL call `reviewApi.approve` with the file ID and employee ID before updating local state
4. WHEN the `REQUEST_REVISION` action is dispatched, THE DocumentWorkflow_Context SHALL call `reviewApi.reject` with the file ID, employee ID, and rejection comments before updating local state
5. WHEN the `ADD_REQUEST` action is dispatched, THE DocumentWorkflow_Context SHALL send the request data to the backend and use the returned server-generated ID in local state
6. IF any API call within DocumentWorkflow_Context fails, THEN THE DocumentWorkflow_Context SHALL set an error state accessible to consuming components and preserve the previous local state

### Requirement 7: Client Dashboard — Live Data Fetching

**User Story:** As a client user, I want the Client Dashboard to display my real document progress from Box, so that I see accurate status of my tax documents.

#### Acceptance Criteria

1. WHEN the Client_Dashboard mounts, THE Client_Dashboard SHALL call `portalApi.getClientProgress` with the authenticated client ID and render the returned documents in the document request list
2. WHILE the `getClientProgress` API call is in flight, THE Client_Dashboard SHALL display a Loading_State (skeleton loaders) in place of the document list and progress tracker
3. IF the `getClientProgress` API call fails, THEN THE Client_Dashboard SHALL display an Error_State with the error message and a "Retry" button that re-invokes the API call
4. WHEN the client progress data is received, THE Client_Dashboard SHALL compute and display the overall completion percentage from the `statusCounts` field in the response
5. THE Client_Dashboard SHALL replace the hardcoded `PREPARER_REQUESTS` array with document requests returned by the `getClientProgress` API response
6. THE Client_Dashboard SHALL replace the hardcoded `TAX_STEPS` progress tracker with status derived from the `statusCounts` in the API response

### Requirement 8: Employee Dashboard — Live Data Fetching

**User Story:** As an employee user, I want the Employee Dashboard to display my real pending reviews and client assignments from Box, so that I see accurate workload data.

#### Acceptance Criteria

1. WHEN the Employee_Dashboard mounts, THE Employee_Dashboard SHALL call `portalApi.getEmployeeDashboard` with the authenticated employee ID and render the returned data
2. WHILE the `getEmployeeDashboard` API call is in flight, THE Employee_Dashboard SHALL display a Loading_State (skeleton loaders) in place of the client list and stat cards
3. IF the `getEmployeeDashboard` API call fails, THEN THE Employee_Dashboard SHALL display an Error_State with the error message and a "Retry" button
4. THE Employee_Dashboard SHALL replace the hardcoded `CLIENTS` array with the `clientChecklists` array from the API response
5. THE Employee_Dashboard SHALL replace the hardcoded stat card values (Assigned Clients, Pending Review, Completed, AI Extractions) with computed values from the API response
6. WHEN the employee selects a document for review, THE Employee_Dashboard SHALL pass the real Box file ID to the Review_Mode component

### Requirement 9: CXO Dashboard — Live Data Fetching

**User Story:** As a CXO user, I want the CXO Dashboard to display real firm-wide portfolio data from Box, so that I see accurate compliance and filing metrics.

#### Acceptance Criteria

1. WHEN the CXO_Dashboard mounts, THE CXO_Dashboard SHALL call `portalApi.getCXOPortfolio` and render the returned portfolio data
2. WHILE the `getCXOPortfolio` API call is in flight, THE CXO_Dashboard SHALL display a Loading_State (skeleton loaders) in place of the portfolio panels and stat cards
3. IF the `getCXOPortfolio` API call fails, THEN THE CXO_Dashboard SHALL display an Error_State with the error message and a "Retry" button
4. THE CXO_Dashboard SHALL replace the hardcoded `FIRMS` array with the `clients` array from the API response, mapping `completionPercentage` to the compliance progress bars
5. THE CXO_Dashboard SHALL replace the hardcoded stat card values (Total Clients, Docs Pending, Avg. Compliance, Overdue Filings) with values from the `firmTotals` field in the API response
6. WHEN the API response includes a `nextCursor`, THE CXO_Dashboard SHALL display a "Load More" control that calls `portalApi.getCXOPortfolio` with the cursor to fetch additional pages

### Requirement 10: Upload Dropzone — Real Box Upload

**User Story:** As a client user, I want file uploads to go to Box via the backend, so that uploaded documents are stored in my secure Box vault.

#### Acceptance Criteria

1. WHEN a file is dropped or selected, THE Upload_Dropzone SHALL call `documentApi.upload` with the file, the vault folder ID from DocumentWorkflow_Context, and the optional request ID
2. WHILE the upload API call is in flight, THE Upload_Dropzone SHALL display the progress bar animating toward 90% and update to 100% when the API call completes
3. WHEN the upload API call completes successfully, THE Upload_Dropzone SHALL pass the returned Box file object (containing `fileId`, `fileName`, `size`) to the `onUpload` callback
4. IF the upload API call fails, THEN THE Upload_Dropzone SHALL reset the progress bar, clear the file name display, and show an alert with the error message
5. IF no `folderId` prop is provided, THEN THE Upload_Dropzone SHALL log an error and prevent the upload from starting

### Requirement 11: Review Mode — Real Approve/Reject/Waive Actions

**User Story:** As an employee user, I want approve, reject, and waive actions in Review Mode to call the real backend, so that review decisions persist to Box metadata.

#### Acceptance Criteria

1. WHEN the employee clicks "Approve" in Review_Mode, THE Review_Mode SHALL call `reviewApi.approve` with the Box file ID and the authenticated employee ID
2. WHEN the employee submits a revision with comments in Review_Mode, THE Review_Mode SHALL call `reviewApi.reject` with the Box file ID, employee ID, and the rejection reason text
3. WHEN the employee clicks "Waive" in Review_Mode, THE Review_Mode SHALL call `reviewApi.waive` with the Box file ID, employee ID, and the waive reason text
4. WHILE any review API call is in flight, THE Review_Mode SHALL disable the Approve, Request Revision, and Waive buttons and display a loading indicator on the active button
5. IF a review API call fails, THEN THE Review_Mode SHALL display an inline error message below the action buttons and re-enable the buttons for retry
6. WHEN a review API call succeeds, THE Review_Mode SHALL update the local document status and navigate back to the document list

### Requirement 12: Request Creator — Persist via Backend

**User Story:** As an employee user, I want document requests created in the Request Creator drawer to persist via the backend, so that clients see real requests in their portal.

#### Acceptance Criteria

1. WHEN the employee submits the Request_Creator form, THE Request_Creator SHALL send the request data (name, description, dueDate, priority, clientId) to the backend API
2. WHEN the backend returns the created request with a server-generated ID, THE Request_Creator SHALL dispatch the `ADD_REQUEST` action with the server response data
3. IF the backend API call fails, THEN THE Request_Creator SHALL display an inline error message above the submit button and keep the form open with the entered data preserved
4. WHILE the create request API call is in flight, THE Request_Creator SHALL disable the submit button and display a loading indicator

### Requirement 13: Auth Context — Box App User Token Integration

**User Story:** As a frontend developer, I want the Auth Context to obtain real Box App User tokens from the backend, so that API calls are authenticated with proper Box credentials.

#### Acceptance Criteria

1. WHEN a user selects a demo role and logs in, THE Auth_Context SHALL call the backend to obtain a Box App User token for the selected role and store the token in state
2. THE Auth_Context SHALL expose the current token via the context value so that the API_Service_Layer can include the token in request headers
3. WHEN the stored token is within 5 minutes of expiry, THE Auth_Context SHALL proactively request a new token from the backend before the current token expires
4. WHEN the user logs out, THE Auth_Context SHALL clear the stored token and user state
5. IF the token request fails, THEN THE Auth_Context SHALL set an error state and allow the user to retry login

### Requirement 14: Loading and Error State UX

**User Story:** As a user, I want clear visual feedback when data is loading or when errors occur, so that I understand the application state at all times.

#### Acceptance Criteria

1. WHILE any dashboard data fetch is in flight, THE corresponding dashboard component SHALL display skeleton loaders matching the layout of the expected content
2. WHEN an API call fails, THE corresponding component SHALL display an error message in a red-tinted glassmorphism panel consistent with the application design system
3. THE error panel SHALL include a "Retry" button that re-invokes the failed API call
4. WHILE a mutating API call (approve, reject, upload, create) is in flight, THE triggering button SHALL display a spinner icon and be disabled to prevent duplicate submissions
5. IF multiple API calls fail in sequence for the same component, THEN THE component SHALL display only the most recent error message to avoid stacking error panels

### Requirement 15: Notification Polling

**User Story:** As a user, I want to see real-time notifications from the backend, so that I am informed of document status changes without refreshing the page.

#### Acceptance Criteria

1. WHEN a user is authenticated, THE frontend SHALL poll `notificationApi.getNotifications` with the user recipient ID at a 30-second interval
2. WHEN new notifications are received, THE frontend SHALL display a notification badge count in the top navigation bar
3. WHEN the user clicks the notification badge, THE frontend SHALL display the list of notifications with event type, message, and timestamp
4. WHEN the user logs out, THE frontend SHALL stop the notification polling interval

### Requirement 16: Preview Token Integration

**User Story:** As an employee reviewing a document, I want to see a real Box Content Preview of the uploaded file, so that I can visually verify the document content.

#### Acceptance Criteria

1. WHEN Review_Mode loads for a document with an uploaded file, THE Review_Mode SHALL call `tokenApi.getPreviewToken` with the file ID and the authenticated user ID
2. WHEN the preview token is received, THE Review_Mode SHALL render a Box Content Preview iframe in the Document Preview Pane using the token and file ID
3. IF the preview token request fails, THEN THE Review_Mode SHALL display a fallback skeleton placeholder with an error message in the Document Preview Pane
4. THE preview token SHALL be refreshed by requesting a new token when the current token is within 5 minutes of expiry
